# SOP_TAKEDOWN.md — 권리침해 신고 임시조치 운영 매뉴얼

> **Write permission**: CEO 단독 저자. 다른 에이전트는 본 문서를 직접 편집하지 않으며, 변경 제안은 KOR-534 또는 후속 티켓 댓글로 제출.
> **법적 근거**: 정보통신망 이용촉진 및 정보보호 등에 관한 법률(이하 "정보통신망법") 제44조의2, 동법 시행령 제13조.
> **연관 티켓**: [KOR-529](/KOR/issues/KOR-529) 제5조, [KOR-530](/KOR/issues/KOR-530), [KOR-534](/KOR/issues/KOR-534).
> **벌칙**: 임시조치 의무 미이행 시 정보통신망법 제74조 제1항 제3호에 따라 처벌 대상이 될 수 있음.

## 1. 목적과 적용 범위

- **적용 대상**: packlinx.com에 게시된 업체 정보(상호, 설명, 카테고리, "비슷한 업체" 추천 슬롯 노출 등) 및 사용자 게시물 일반에 대한 권리침해 신고
- **적용 범위**:
  - 명예훼손/신용훼손/업무방해(형법 제307조·제313조·제314조)
  - 부정경쟁(부정경쟁방지법 제2조 제1호)
  - 표시광고법 위반(제3조)
  - 사실 오기·부당한 노출 일반
- **법적 근거**: 정보통신망법 제44조의2 제2항(임시조치 의무) + 시행령 제13조(임시조치 기간 30일 이내)
- **본 SOP의 위치**: KOR-529 제5조(분쟁 발생 시 임시조치 SLA)의 운영 책임 이행 문서

## 2. 접수 채널 (Intake Channel)

### 2-1. 1순위 채널
- **이메일**: `legal@pkging.kr` *(신설 — 정보통신망법 신고 전용 메일박스)*
- **웹폼**: `/opt-out` 페이지의 "권리침해 신고 (정보통신망법 임시조치 요청)" 옵션 *(KOR-??? 후속 티켓에서 `request_type='takedown'` 추가)*

### 2-2. 대체 채널 (자동 라우팅)
- `privacy@pkging.kr` 또는 사이트 일반 문의 메일로 도착한 권리침해 신고는 운영자가 영업시간 기준 4시간 이내 `legal@pkging.kr`로 전달, 접수 시각은 최초 도착 시각 기준
- 신고자는 접수 채널을 모르더라도 `/opt-out` 또는 푸터 링크를 통해 접근 가능해야 함

### 2-3. 노출 위치 (반드시 게시)
- 사이트 푸터: "권리침해 신고" 링크 → `/opt-out` (또는 별도 `/notice` 페이지)
- 이용약관(`/terms`): 신설 "제X조 (권리침해 신고 및 임시조치)" 조항에 명시
- 개인정보처리방침(`/privacy`): "권리침해 신고 처리" 항목 추가, 보존 기간 명시(§9 참조)
- 노출 의무 근거: 정보통신망법 제44조의2 제2항 — 신고 절차를 정보통신서비스 이용자가 알 수 있도록 안내해야 함

## 3. 신고 양식 (필수 정보)

신고가 유효하려면 다음 정보가 모두 포함되어야 함. 누락 시 §5-1 보완 요청 절차 적용.

1. **신고자 신원**: 성명/상호 (대리인 신고 시 대리권 입증 자료)
2. **신고자 연락처**: 이메일(필수). 통지를 받지 못하면 SLA 산정 불가
3. **신고 대상 콘텐츠 URL**: 예) `https://packlinx.com/companies/{slug}` — "비슷한 업체" 슬롯 신고 시 슬롯 위치도 명시
4. **권리침해 사유**: 명예훼손 / 영업방해 / 부정경쟁 / 표시광고법 위반 / 사실 오기 / 기타 중 택일 + 구체 설명
5. **권리 입증 자료**(선택): 등기부등본, 인증서, 권리 증명 자료 등

## 4. SLA 타임라인

| 단계 | 시한 | 책임자 |
|------|------|--------|
| 접수 확인 (자동/수동) | 영업시간 기준 4시간 이내 | CS 운영자 |
| 임시조치 (비공개 처리) | 영업시간 기준 **24시간 이내** | CS 운영자 → CTO/Backend |
| 양당사자 의견 청취 시작 | 임시조치 후 24시간 이내 | Legal Counsel |
| 종결 결정 (영구 비공개 / 노출 복원 / 부분 복원) | 임시조치 후 **30일 이내** | Legal Counsel + CEO |
| 처리 결과 통지 | 종결 결정 후 24시간 이내 | CS 운영자 |

> **영업시간 정의**: 평일 09:00–18:00 KST. 비영업시간 접수는 차회 영업일 시작 시점부터 SLA 카운트. 단, 명예훼손/형사 책임 가능 사안은 비영업시간이라도 즉시 CTO 호출(§11 참조).

## 5. 처리 절차

### 5-1. 접수 확인
1. 운영자가 신고 메일/폼 접수를 확인 → `takedown_log` 시트(또는 운영 도구)에 신규 레코드 생성, `case_id = TKD-{YYYYMMDD-XXXX}` 부여
2. 자동/수동 회신: §7-1 "접수 확인" 템플릿 발송
3. §3 필수 정보 누락 시 24시간 내 보완 요청 메일 발송. 보완 요청 후 신고자가 7일 내 미응답 시 자동 종결 + 종결 통지(§7-4)
4. 운영자는 접수 시점에 §11 "악의적 반복 신고" 체크리스트 확인

### 5-2. 임시조치
**Tier A — 즉시 가용 (migration 045 배포 후)**
- 컬럼: `companies.similar_optout_at` (timestamptz, nullable) — migration 045_similar_optout.sql에서 도입
- `get_similar_companies()` 함수가 `c.similar_optout_at IS NULL` 필터 적용 → "비슷한 업체" 슬롯에서만 비공개, 페이지 자체는 유지
- 적용 SQL: `UPDATE companies SET similar_optout_at = NOW() WHERE id = '<uuid>';`
- REST API: `PATCH /api/companies/[id]/similar-optout` (업체 소유자 인증) — 운영자 직접 토글 시 service-role SQL 사용

**Tier B — 즉시 가용 (KOR-352에서 도입된 `is_hidden` 활용)**
- `companies.is_hidden = TRUE` → 모든 공개 리스팅에서 비공개(소프트 삭제)
- 단점: "비슷한 업체 노출만" 임시조치하기에는 오버슈팅이나, 정보통신망법 SLA를 충족하는 1라인 SQL/어드민 조치
- 적용 SQL: `UPDATE companies SET is_hidden = TRUE WHERE id = '<uuid>';` (RLS 정책이 자동으로 공개 리스팅에서 제외)

**적용 분기 규칙**
- 신고 사유가 "비슷한 업체 노출만 부당"인 경우 → Tier A 즉시 적용 (페이지 자체는 유지되므로 신고자 동의 단계 불필요, SLA 시간 단축)
- 신고 사유가 "페이지 전체 노출 부당"(명예훼손, 사실 오기 등)인 경우 → Tier B 즉시 적용
- 임시조치 시 `takedown_log`에 (a) 적용 시각, (b) 적용자, (c) 적용 SQL/어드민 조치, (d) Tier 기록(§8 참조)

### 5-3. 양당사자 의견 청취
- **신고자에게**: §7-2 "임시조치 적용 통지" — 30일 내 의견 보강 가능 안내
- **피신고 업체에게**: 등록된 업체 연락처(이메일)로 §7-3 "권리침해 신고 접수 및 임시조치 통지" 발송
  - 14일 이내 의견 회신 요청
  - **회사 모토(KOR-371) 준수**: 피신고 업체 연락처는 가입/등록 시 수집된 이메일만 사용. 전화/주소를 통한 접촉 금지
  - 등록 이메일이 없는 업체(외부 출처 데이터)는 등기부등본·홈페이지 공개 이메일 등 공개 채널을 통해 1회 시도, 그 외에는 의견청취 생략 + 사유 로그
- 양 당사자 답변 수령 시 `takedown_log`에 첨부 (URL 또는 메일 보관 ID)

### 5-4. 종결 결정
- Legal Counsel이 양당사자 자료를 검토 후 다음 중 하나로 초안 결정, CEO 승인:
  - **영구 비공개**: 권리침해 입증, 임시조치 유지
  - **노출 복원**: 권리침해 미인정 또는 불충분, `is_hidden = FALSE` 또는 `excluded_from_similar = FALSE`로 복원
  - **부분 복원**: 페이지 일부 수정(예: 잘못된 설명 정정) 후 노출 복원
- 결정 시한: 임시조치 후 30일 이내 (시행령 제13조 한계). 30일 도과 시 임시조치 유지가 위법성을 띌 수 있으므로, **D-25일 알림 + D-29일 CEO 에스컬레이션**으로 안전선 확보
- 의견청취 결과가 모호한 경우 영구 비공개 — 신고자 측 권리 보호 우선(과실 책임 방어 가능)

### 5-5. 결과 통지
- 신고자·피신고자 양쪽에 §7-4 "종결 통지" 발송
- 처리 결과를 `takedown_log`에 기록, `decision`, `decision_at`, `closed_at` 입력
- 종결 후 §9 "증빙 보존 정책"에 따라 3년간 보존

## 6. 임시조치 도구 (어드민/DB)

### 6-1. 즉시 가용 (CTO 확인 사항)
- `UPDATE companies SET is_hidden = TRUE WHERE id = '<uuid>';` (Tier B)
  - 검증 필요: service-role SQL 직접 실행 권한 → CTO 또는 Backend 운영자만 실행
  - RLS 정책 `companies_public_read`이 `is_hidden = FALSE` 조건을 가지므로 즉시 효력 (KOR-352 migration 036 확인 완료)

### 6-2. 배포 완료 후 즉시 가용 (Tier A)
- `companies.similar_optout_at` (timestamptz) — migration 045_similar_optout.sql에 구현 완료, 배포만 진행하면 가용
- `get_similar_companies()` 함수에 `c.similar_optout_at IS NULL` 필터 적용 완료
- REST API `PATCH /api/companies/[id]/similar-optout` 가용 (업체 소유자 인증)

### 6-3. 향후 확보 권장 (별도 후속 티켓 발주)
- 어드민 대시보드에 "임시조치 토글" UI 추가 (운영자가 SQL 직접 실행하지 않도록)
- 임시조치 만료(30일 경과) 자동 복원 cron 또는 수동 점검 루틴

## 8. 처리 로그 양식 (`takedown_log`)

운영 초기에는 Google Sheets 또는 Notion 데이터베이스로 운영. KOR-??? 후속 티켓에서 DB 테이블화 권장.

| 필드 | 타입 | 설명 |
|------|------|------|
| `case_id` | string | `TKD-YYYYMMDD-XXXX` 형식 |
| `received_at` | timestamp (KST) | 접수 시각 |
| `received_via` | enum | `email` / `form` / `other` |
| `reporter_name` | string | 신고자 성명/상호 |
| `reporter_email` | string | 신고자 이메일 |
| `target_url` | string | 신고 대상 페이지 URL |
| `target_company_id` | uuid | `companies.id` (있으면) |
| `claim_type` | enum | `defamation` / `business_interference` / `unfair_competition` / `false_advertising` / `factual_error` / `other` |
| `claim_summary` | text | 신고 사유 요약 |
| `interim_action_at` | timestamp | 임시조치 적용 시각 |
| `interim_action_tier` | enum | `A_similar_optout` / `B_is_hidden` / `other` |
| `interim_action_actor` | string | 적용 운영자 ID |
| `interim_action_sql` | text | 적용한 SQL 또는 어드민 조치 기록 |
| `reporter_evidence_url` | string[] | 신고자 제출 증빙 (메일 링크/파일 ID) |
| `respondent_notified_at` | timestamp | 피신고 업체 통지 시각 |
| `respondent_notified_method` | enum | `registered_email` / `public_email` / `not_reachable` |
| `respondent_response` | text | 피신고 업체 답변 요약 |
| `decision` | enum | `permanent_hide` / `restore` / `partial_restore` / `auto_close_no_response` |
| `decision_at` | timestamp | 결정 시각 |
| `decision_reason` | text | 결정 사유 |
| `closed_at` | timestamp | 종결 통지 시각 |
| `retention_until` | date | 증빙 보존 만료일 = `closed_at` + 3년 |

## 9. 증빙 보존 정책

- **보존 기간**: 종결 후 **3년**
  - 근거1: 정보통신망법상 명문 보존 의무는 없음
  - 근거2: 민법 제766조 제1항(불법행위 손해배상 청구권 단기 시효 3년) — 분쟁 대응 자료 안전선
  - 근거3: 상법 제64조(상사채권 5년 시효)는 일반 상사 거래 기준이므로, 본 SOP에서는 단기 시효 3년을 채택
- **보존 대상**: `takedown_log` 전체 + 신고 메일 원본 + 피신고자 회신 메일 + 적용 SQL 로그
- **보존 사유의 법적 정당화**: 「개인정보보호법」 제15조 제1항 제6호(정보주체의 권리보다 명백히 우선하는 정당한 이익) — 사후 분쟁 대응
- **3년 경과 후 처리**: 자동 파기 또는 익명화 (운영팀 분기 점검). PIPA 제21조(목적 달성 시 즉시 파기) 정합성 확보를 위해 만료 알림 자동화 권장
- **개인정보처리방침 정합성**: 본 SOP 채택 시 `/privacy` 페이지에 "권리침해 신고 처리 — 보존 3년" 항목 추가 (별도 티켓)

- **신고자 신원 미확인**: 익명 신고는 원칙적으로 임시조치 보류 후 신원 보완 요청. 단, 명백한 공익 침해(범죄 의심 등)는 Legal Counsel 직접 판단으로 임시조치 가능
